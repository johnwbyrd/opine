OPINE

Floating point is a set of opinions.
This library lets you hold your own.

github.com/johnwbyrd/opine

Welcome. This is a talk about floating-point numbers, but I promise it is not a talk about IEEE 754 trivia. It's a talk about a C++ library called OPINE, and the idea behind the name: every floating-point format ever designed is really just a bundle of opinions — and this library lets you pick your own. Mostly I'm going to show you things rather than explain things. Every output on these slides is a real program run, unedited.

Floating point used to be simple

float   double

Then machine learning happened:

bfloat16 · float16 · FP8-E5M2 ·
FP8-E4M3 · FP8-E4M3FNUZ ...

...and it never was simple: x87 80-bit · IBM hex floats · binary128/256 · posits · ...

For most of our careers, floating point meant two types. That world is over. ML hardware runs on 16-bit and even 8-bit floats — and there are at least three incompatible 8-bit formats in production silicon right now. Look backwards and it was never clean either: Intel's 80-bit format is still inside every x86 chip, IBM mainframes did something completely different, and researchers keep inventing new encodings. The problem: each of these is an island. Different bit layouts, different NaNs, different overflow behavior — and usually no trustworthy software implementation to test against.

Same questions.
Different answers.

Strip away the details and every float format is answers to the same handful of questions. How many bits of precision is worth paying for? Does NaN deserve bit patterns? Is there a negative zero? When a result doesn't fit, do you go to infinity or stick at the biggest value? Which direction do leftover digits round? IEEE 754 is one magnificent, battle-tested set of answers — but it IS a set of answers, written by a committee, not a law of nature. FP8-E4M3FNUZ answers "no negative zero, that pattern is NaN." A GPU fast mode answers "denormals are zero, don't care." OPINE's pitch: make the questions the API. Every answer is a template parameter, and every format is the same machinery with different settings.

Hello, OPINE


#include <opine/opine.hpp>
using namespace opine;

using f32 = float32;   // IEEE binary32

auto x = fromNative<f32>(1.5f);
auto y = fromNative<f32>(2.25f);
auto z = add<f32>(x, y);
  
3.75
The whole API in one slide. Header-only, one include. You pick a format — here plain binary32. fromNative gets a value in, add does arithmetic IN that format, toFloat gets it back out for printing. One thing worth noticing: operations work on bit patterns, not on a wrapped number class. That's deliberate. The formats we'll use in a minute don't exist in your CPU, so nothing can be allowed to silently fall back to hardware floats. When every operation names its format, you always know exactly which arithmetic happened.

Your computer has been
lying to you


auto x = fromString<float64>("0.1");
puts(toString<float64>(x, 60).c_str());
  
0.1000000000000000055511151231257827021181583404541015625
Here's a demo for everyone who has ever printed 0.1 and gotten 0.1 back. No binary format can store one-tenth. Your double stores its nearest neighbor, and your language's print function rounds it back to "0.1" to be polite. OPINE's toString will print the stored value EXACTLY — every one of those 55 digits is true. That's what is actually in the variable. This is not a bug and not sloppiness — it's a tradeoff someone chose. But you should get to SEE it.

...and "0.1" in smaller formats

fp8_e4m3  0.1015625

An 8-bit float misses by 1.5% before you do any math at all.

Same "0.1", smaller formats. In an 8-bit float the nearest storable value is 0.1015625 — one and a half percent off before you've done a single operation. If you're quantizing a model to FP8, these are the values your model actually sees. Now you can look at them, exactly, instead of guessing.

π in your pocket formats

fp8_e5m2      3           -4.5%
fp8_e4m3      3.25        +3.5%
fp8_e4m3fnuz  3.25        +3.5%
bfloat16      3.140625    -0.031%
float32       3.14159274  +0.0000028%
Same value, five formats, one conversion each. Pi in FP8-E5M2 is just... 3. In E4M3 it's 3.25. A fun detail hiding in the middle: the two E4M3 variants show the same VALUE here, but they store different BITS for it — 0x45 versus 0x4D — because they disagree about how the encoding works. That's exactly the kind of detail that burns you when tensors move between vendors. This table is the honest answer to "how much accuracy do I give up at each width" — measured, not estimated, because the arithmetic is bit-exact for every format on the slide.

Invent a float before lunch


// 12 bits: 1 sign + 5 exp + 6 fraction
using MyFp12 =
    Type<numbers::IEEE754<5, 6>,
         layouts::IEEE<5, 6, true>>;

auto x = fromNative<MyFp12>(3.14159f);
  
3.15625
Suppose FP8 is too coarse for your data and FP16 is wasteful. Here's a 12-bit float — one using-declaration, and there's pi living in it. It is not a second-class citizen: add, multiply, square root, fused multiply-add, conversion to and from every other format, exact decimal printing — everything works on it the moment it's declared, because it's the same pipeline as the standard formats with different numbers plugged in. If you design accelerator hardware, or evaluate people who do, this is prototyping a number system before silicon exists. And the compiler specializes everything at compile time — there's no interpreter in the loop.

Sort floats as plain integers


// a two's-complement FLOAT encoding
using rbj8 = RbjType<4, 3>;

std::sort((int8_t*)begin,   // !!
          (int8_t*)end);
  
before  1.0 -1.0 .5 -.5 2.5 -2.5
after  -2.5 -1.0 -.5 .5 1.0 2.5 ✓
Now a party trick with a real point. This format — an idea from the DSP world, and how machines like the PDP-10 did it decades ago — encodes floats in two's complement, like ordinary signed integers. The consequence: integer order IS float order. This array holds FLOAT values, and I'm sorting it as raw signed bytes — and it comes out numerically sorted. Do the same with IEEE floats and the negatives come out backwards, because IEEE is sign-magnitude. Why show this? Because it's a LAYOUT opinion with a superpower — think radix sort, think hardware that only has integer comparators. IEEE chose differently, for good reasons. But it was a choice — and in OPINE, choices are template parameters. This encoding gets the same arithmetic and the same tests as everything else.

One addition, six answers

1.0 + half a last-digit — a perfect tie

TiesToEven   1.0         ← default
TiesAway     1.00000012
TowardZero   1.0
TowardPos    1.00000012
TowardNeg    1.0
ToOdd        1.00000012
Rounding. When a result lands exactly halfway between two representable values, somebody has to pick a side. Here is the same addition under all six rounding modes — six different opinions, six answers, and every one is "correct" by its own rules. The default, ties-to-even, rounds DOWN here because 1.0's last bit is even. Ties-away rounds up. The directed modes go where they're pointed. And round-to-odd at the bottom is the specialist's tool — it makes intermediate results safe to round a second time. If you've never needed it, ignore it; if you have, you know how rare it is to find it implemented and tested.

The mode is part of the type


using f32_up =
    Type<numbers::IEEE754<8, 23>,
         layouts::IEEE<8, 23, true>,
         rounding::TowardPositive>;

No global state. Two modes can coexist
in one program, statically.

On a CPU, the rounding mode is a global control register — changing it is a whole-program side effect, and library code that flips it and forgets to restore it causes legendary bugs. In OPINE the mode is a template parameter: it's part of the type, two modes coexist in one binary, and nothing can leak. Interval arithmetic — where you compute every bound twice, once rounding down and once rounding up — falls out naturally: they're just two types.

When the answer doesn't fit

128 × 512 in FP8 — max finite is 57344

ToNearest    +inf   ← off the end
TowardZero   57344  ← saturate
ToOdd        57344
Overflow is another opinion. The IEEE rule is subtler than most people remember: an overflowing result becomes infinity only when the rounding mode would carry you upward. Round toward zero pins you at the biggest finite value instead. And formats without an infinity encoding — like the FNUZ FP8 — always saturate, in every mode. In an 8-bit format whose whole number line ends at 57344, you hit this constantly. Which raises the real question — how do you find out it happened? Next slide.

...and the result tells you


// every op returns {bits, flags}
auto r = mul<E5M2Status>(a, b);

if (r.flags & FlagOverflow)
  // you KNOW. no guessing.
  
57344   overflow=1  inexact=1
How errors are reported is a policy too — the third axis. Silent, which is the default. Sticky flags you can poll, like a CPU. Or my favorite, shown here: every operation returns its result bits TOGETHER with its flags. Same multiply as the last slide — the saturated value comes back carrying "overflow happened, and the result is inexact." Saturation without a flag is silent data corruption. Saturation WITH a flag is a diagnosable event. That one-word difference is a debugging session you don't have.

The answer multiplication erases

(1+ε)² − (1+2ε) = ε²   ...algebraically


auto u = add<f32>(mul<f32>(a, a), c);
auto f = fma<f32>(a, a, c);
  
mul then add:  0
fma:           1.42108547e-14  = ε²
Fused multiply-add — the operation modern hardware is built around. This expression equals epsilon squared: a tiny but perfectly real number. Compute it the obvious way and you get zero — the multiply must round before the subtraction, and epsilon squared is exactly the part that rounds away. The fused version keeps the whole double-width product alive until one single rounding at the end. There's the true answer. This isn't a curiosity. fma is the engine inside dot products, polynomial evaluation, and every compensated-arithmetic trick in the numerical toolbox. OPINE's is genuinely fused — one rounding — in every format, including formats no hardware fma has ever heard of.

Turn the precision knob

Rump's polynomial (1988) — true answer ≈ −0.8274

float32    -6.3e+29   wrong
float64    -1.2e+21   wrong
float128    1.1726    wrong SIGN
float256   -0.82740   correct
float1024  -0.82740   correct

Same source code. Only the type changes.

This polynomial is famous. In double precision it returns minus a billion trillion — with total confidence, no warning. That's not an exotic bug; it's catastrophic cancellation, and it hides in ordinary-looking arithmetic. Here's the experiment nothing else lets you run this easily: the same code at every precision. Garbage at 32 bits. Garbage at 64. At 128 bits it has the wrong SIGN — arguably the scariest row on the slide. At 256 bits, the truth finally appears. If you suspect numerical instability in real code, this is the diagnostic: crank the knob and watch where your answer stops changing. And because the wide formats run the same verified pipeline, float256 isn't a different library with different semantics — it's the same opinions, wider.

The ladder goes all the way up: IEEE's formula defines a binary1024 format, so OPINE implements it — 997 bits of precision, about 300 decimal digits. Here's pi computed with nothing but add, subtract, multiply, divide, in that format — this slide shows the first 160-odd digits, and every single one is correct, including the printing: the decimal conversion is correctly rounded at every width, so turning bits into digits loses nothing. There is no size ceiling in the library — arithmetic runs on arrays of machine words sized to the job. It's not the fastest bignum in the world and isn't trying to be: it's the SAME verified pipeline as the 8-bit formats, just wider. One library, 8 bits to 1024.

Why believe any of this?

every · possible · input

8-bit formats: all 65,536 input pairs ×
every op × every encoding × every mode,
bits and flags, vs an independent MPFR oracle.

Plus Berkeley SoftFloat and TestFloat —
what FPU vendors use to qualify silicon.

Fair question, because floating-point code is famously easy to get subtly wrong. This is the part of the project I'd defend hardest. Small formats get EXHAUSTIVE verification — every one of the 65,536 possible input pairs, for every operation, every encoding, and every rounding mode; it multiplies out to millions of comparisons, each checked bit-for-bit against a reference built on MPFR, the arbitrary-precision library the numerical community trusts. The oracle shares no code with the library. It has caught real bugs in the library, and the library has caught real bugs in the oracle — that cross-pressure is the design. IEEE formats additionally run against Berkeley SoftFloat and through Berkeley TestFloat, the same conformance suite hardware FPU designers use. Wider formats get structured plus randomized batteries up through binary1024. When an earlier slide said "exactly," this is why the word was safe.

The takeaway: floating point is not a fact of nature you have to accept — it's a set of engineering tradeoffs someone chose for you. Usually they chose well! But the zoo is here: FP8 in your models, bfloat16 in your accelerators, maybe a custom format in your next chip. OPINE gives you one place where all of them — the careful ones, the sloppy-on-purpose ones, the ones you invent — are complete, verified, and side by side. Go have opinions about your floats.
