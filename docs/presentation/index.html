OPINE

Floating point is a set of opinions.
This library lets you hold your own.

github.com/johnwbyrd/opine — header-only C++20

Welcome. This is a talk about floating-point numbers, but I promise it is not a talk about IEEE 754 trivia. It's a talk about a C++ library called OPINE, and the idea behind the name: every floating-point format ever designed is really just a bundle of opinions — and this library lets you pick your own. Mostly I'm going to show you things rather than explain things. Every output on these slides is a real program run, unedited.

Floating point used to be simple

float  and  double

Then machine learning happened:

bfloat16   float16   FP8-E5M2   FP8-E4M3   FP8-E4M3FNUZ ...

And it was never actually simple:

x87 80-bit extended  ·  IBM hex floats  ·  CDC one's-complement  ·  binary128/256  ·  posits  ·  ...

For most of our careers, floating point meant two types. That world is over. ML hardware runs on 16-bit and even 8-bit floats — and there are at least three incompatible 8-bit formats in production silicon right now. Look backwards and it was never clean either: Intel's 80-bit format is still inside every x86 chip, IBM mainframes did something completely different, and researchers keep inventing new encodings. The problem: each of these is an island. Different bit layouts, different NaNs, different overflow behavior — and usually no trustworthy software implementation to test against.

Same questions, different answers:
How much precision? Does NaN exist? Negative zero? What happens on overflow? Which way do we round?

Strip away the details and every float format is answers to the same handful of questions. How many bits of precision is worth paying for? Does NaN deserve bit patterns? Is there a negative zero? When a result doesn't fit, do you go to infinity or stick at the biggest value? Which direction do leftover digits round? IEEE 754 is one magnificent, battle-tested set of answers — but it IS a set of answers, written by a committee, not a law of nature. FP8-E4M3FNUZ answers "no negative zero, that pattern is NaN." A GPU fast mode answers "denormals are zero, don't care." OPINE's pitch: make the questions the API. Every answer is a template parameter, and every format is the same machinery with different settings.

Hello, OPINE


#include <opine/opine.hpp>
using namespace opine;

using f32 = float32;                    // IEEE binary32

auto x = fromNative<f32>(1.5f);
auto y = fromNative<f32>(2.25f);
auto z = add<f32>(x, y);

printf("%g\n", double(toFloat<f32>(z)));
  
3.75
The whole API in one slide. Header-only, one include. You pick a format — here plain binary32. fromNative gets a value in, add does arithmetic IN that format, toFloat gets it back out. One thing worth noticing: operations work on bit patterns, not on a wrapped number class. That's deliberate. The formats we'll use in a minute don't exist in your CPU, so nothing can be allowed to silently fall back to hardware floats. When every operation names its format, you always know exactly which arithmetic happened.

Your computer has been lying to you


auto x = fromString<float64>("0.1");
puts(toString<float64>(x, 60).c_str());
  
0.1000000000000000055511151231257827021181583404541015625

...and in other formats:

fp8_e4m3   0.1015625
float16    0.0999755859375
float32    0.100000001490116119384765625
Here's a demo for everyone who has ever printed 0.1 and gotten 0.1 back. No binary format can store one-tenth. Each one stores its own nearest neighbor, and your language's print function rounds it back to "0.1" to be polite. OPINE's toString will print the stored value EXACTLY — every one of those 55 digits is true. That's what your double actually contains. And the second block: the same "0.1" in an 8-bit float is 0.1015625 — off by one and a half percent before you've done any math at all. If you're quantizing a model to FP8, these are the values your model actually sees. Now you can look at them.

bfloat16     bits=0x4049  value=3.140625    relerr=-3.1e-04

float32      bits=0x40490FDB  value=3.141592741  relerr=+2.8e-08
Same value, six formats, one line of code each. Pi in FP8-E5M2 is just... 3. In E4M3 it's 3.25. Notice the two E4M3 variants store different BITS — 0x45 versus 0x4D — for the same value, because they disagree about how the encoding works. That's the kind of detail that burns you when you're moving tensors between vendors. This table is the honest version of "how much accuracy do I give up at each width" — measured, not estimated, because the arithmetic is bit-exact for every one of these formats.

Invent a float before lunch


// 12 bits: 1 sign + 5 exponent + 6 fraction. Why not?
using MyFp12 = Type<numbers::IEEE754<5, 6>,
                    layouts::IEEE<5, 6, true>>;

auto x = fromNative<MyFp12>(3.14159f);
puts(toString<MyFp12>(x, 6).c_str());
  
3.15625

Every operation, every conversion, string I/O, the whole test battery — all of it works on MyFp12, immediately.

Suppose FP8 is too coarse for your data and FP16 is wasteful. Here's a 12-bit float — one using-declaration. It is not a second-class citizen: add, multiply, square root, fused multiply-add, conversion to and from every other format, exact decimal printing — everything works on it the moment it's declared, because it's the same pipeline as the standard formats with different numbers plugged in. If you design accelerator hardware, or you're evaluating someone who does, this is prototyping a number system before silicon exists. The compiler specializes everything at compile time; there's no interpreter in the loop.

using rbj8 = RbjType<4, 3>;   // two's-complement float encoding

std::sort(reinterpret_cast<int8_t*>(begin),

== rbj FP8 (two's complement) ==
 before   1.0  -1.0  0.0  0.5  -0.5  2.5  -2.5  4.0  -4.0
 after   -4.0  -2.5  -1.0  -0.5  0.0  0.5  1.0  2.5  4.0   ✓

== IEEE FP8 (sign-magnitude) ==
 after   -0.5  -1.0  -2.5  -4.0  0.0  0.5  1.0  2.5  4.0   ✗
Now for a party trick with a real point. This format — an idea from the DSP world, and how the PDP-10 did it decades ago — encodes floats in two's complement, like ordinary signed integers. The consequence: integer order IS float order. You can std::sort float data as raw bytes and it comes out numerically sorted. Try that with IEEE floats — bottom block — and the negatives come out backwards, because IEEE uses sign-magnitude. Why show this? Because it's a LAYOUT opinion with a superpower — think radix sort, think comparing floats in hardware that only has integer comparators. IEEE chose differently for good reasons. But it was a choice, and in OPINE, choices are template parameters. This encoding is fully supported: same arithmetic, same tests, same everything.

One addition, six answers

1.0 + (half a last-digit) — exactly halfway between two representable values

ToNearestTiesToEven   1.0          ← the default: tie goes to even
ToNearestTiesAway     1.00000012   ← tie rounds up
TowardZero            1.0
TowardPositive        1.00000012
TowardNegative        1.0
ToOdd                 1.00000012   ← always jam the last bit

The mode is a template parameter:


using f32_up = Type<numbers::IEEE754<8, 23>,
                    layouts::IEEE<8, 23, true>,
                    rounding::TowardPositive>;
Rounding. When a result lands exactly halfway between two representable values, somebody has to pick a side. Here is the same addition under all six rounding modes — six different opinions, six answers, and every one of them is "correct" by its own rules. Your CPU supports four of these, and switching them at runtime is a global-state headache. In OPINE the mode is part of the type, so two modes can coexist in one program, statically, with no flags to leak. The weird one at the bottom, round-to-odd, is a specialist's tool: it makes intermediate results safe to round a second time. If you've never needed it, ignore it. If you have — you know how rare it is to find it implemented and tested.

When the answer doesn't fit

128 × 512 in FP8-E5M2 (max finite value: 57344)

ToNearestTiesToEven   +inf        ← round up, off the end
TowardZero             57344      ← saturate at the edge
ToOdd                  57344

// Or make every operation report what happened:
auto r = mul<E5M2Status>(a, b);       // returns {bits, flags}
if (r.flags & FlagOverflow) { /* you KNOW */ }
  
bits 0x7B (57344)   overflow=1  inexact=1
Overflow is another opinion. The IEEE rule is subtle: an overflowing result becomes infinity only when the rounding mode would carry you upward — round toward zero pins you at the biggest finite value instead. And formats without an infinity encoding, like the FNUZ FP8, always saturate. The second half matters more in practice: how do you FIND OUT it happened? OPINE makes that a policy too. Silent — the default. Sticky flags you can poll. Or, my favorite, every operation returns its result bits together with its flags. Saturation without a flag is silent data corruption; saturation WITH a flag is a diagnosable event. In FP8, where the whole number line ends at 57344, you overflow a lot — you want to know.

The answer multiplication erases

(1+ε)² − (1+2ε) = ε²   ...with ε one unit in the last place of 1.0


auto unfused = add<f32>(mul<f32>(a, a), minus_b);
auto fused   = fma<f32>(a, a, minus_b);
  
mul then add : 0
fma          : 1.42108547e-14      (= ε², exactly)
Fused multiply-add, the operation modern hardware is built around. Algebraically this expression equals epsilon squared — a tiny but perfectly real number. Compute it the obvious way and you get zero: the multiply has to round its result before the subtraction, and epsilon squared is exactly the part that rounds away. The fused version keeps the entire double-width product alive until one single rounding at the end — and there's the true answer. This isn't a curiosity: fma is the engine inside dot products, polynomial evaluation, and every compensated-arithmetic trick in the numerical toolbox. OPINE's is genuinely fused — one rounding — in every format, including formats no hardware fma has ever heard of.

Turn the precision knob

Rump's polynomial (1988) at x=77617, y=33096 — true answer ≈ −0.8274

float32     -6.34e+29          wrong magnitude
float64     -1.18e+21          wrong magnitude
float128     1.1726            wrong SIGN
float256    -0.8273960599      correct — finally
float1024   -0.8273960599      correct

Same source code. The only change is the type.

This polynomial is famous. In double precision it returns minus a billion trillion — with total confidence, no warning. This isn't an exotic bug; it's catastrophic cancellation, and it lives in ordinary-looking arithmetic. Here's the experiment nothing else lets you run this easily: the same code at every precision. Watch the answer: garbage at 32 bits, garbage at 64, at 128 bits it has the wrong SIGN — arguably the scariest row — and at 256 bits the truth finally appears. If you suspect a numerical stability problem in real code, this is the diagnostic: crank the knob and see where your answer stops changing. And because these wide formats run the same verified pipeline, float256 isn't a different library with different semantics. It's the same opinions, wider.

The ladder goes all the way up: IEEE's formula defines a binary1024 format, so OPINE implements it — 997 bits of precision, about 300 decimal digits. Here's pi computed with nothing but add, subtract, multiply, divide, in that format. And printing 300 digits is its own achievement: the decimal conversion is correctly rounded at every width, so every digit on this slide is genuinely a digit of pi. There is no size ceiling in the library — arithmetic runs on arrays of machine words sized to the job, on any supported compiler. Not the fastest bignum in the world, and not trying to be: it's the same verified pipeline as the 8-bit formats, just wider. One library, 8 to 1024 bits.

Why believe any of this?

every · possible · input

For 8-bit formats, the tests don't sample: all 256 × 256 = 65,536 input pairs, for every operation, every encoding, every rounding mode, bits and flags — millions of checks against an independent MPFR-based oracle.

Plus second and third opinions: Berkeley SoftFloat and Berkeley TestFloat — the same conformance suite hardware FPU vendors run.

Fair question, because floating-point code is famously easy to get subtly wrong. The answer is the part of the project I'd defend hardest. Small formats get EXHAUSTIVE verification — every one of the 65,536 possible input pairs, for every operation, every encoding, and every rounding mode, which multiplies out to millions of comparisons — each one checked bit-for-bit against a reference built on MPFR, the arbitrary-precision library the numerical community trusts. The oracle shares no code with the library — it has caught real bugs in the library, and the library has caught bugs in the oracle. That cross-pressure is the design. IEEE formats additionally run against Berkeley SoftFloat and through Berkeley TestFloat, which is what actual FPU designers use to qualify silicon. Wider formats get structured and randomized batteries up through binary1024. When a slide earlier said "exactly," this is why the word was safe to use.

The takeaway: floating point is not a fact of nature you have to accept — it's a set of engineering tradeoffs someone chose for you. Usually they chose well! But now the zoo is here: FP8 in your models, bfloat16 in your accelerators, maybe a custom format in your next chip. OPINE gives you one place where all of them — the careful ones, the sloppy-on-purpose ones, the ones you invent — are complete, verified, and side by side. Go have opinions about your floats.
